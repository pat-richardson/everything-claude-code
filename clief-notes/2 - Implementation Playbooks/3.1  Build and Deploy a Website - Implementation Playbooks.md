3.1 Build and Deploy a Website

29:22

### What This Lesson Covers

Full walkthrough of building a real website using Claude Code and deploying it for free on GitHub Pages. Starting from an existing site as reference, ending with a live URL anyone can visit. Total prompts used: under 10.

This lesson shows the complete process. Lessons 3.2 and 3.3 go deeper on the individual pieces.

---

### The Process at a Glance

1. Analyze the reference website in Claude chat
2. Generate a markdown briefing document for Claude Code
3. Set up a workspace folder on your desktop
4. Write the first prompt with structure, scope, and alignment questions
5. Answer Claude's questions and request a PRD
6. Let Claude Code build the first version
7. Review and give design feedback
8. Push to GitHub and deploy on Pages
9. Fix any pathing or deployment issues
10. Confirm the live site works

---

### Key Moments in the Video

**Using Claude chat first, not Claude Code (1:00)**

The build starts in the browser, not the terminal. Claude chat explores the reference website, clicks through pages, and builds an understanding of the structure. This is context gathering. Claude Code is for building. Claude chat is for thinking and analysis.

**The markdown handoff (3:15)**

Claude chat creates a markdown file describing the site structure. This file transfers context from one Claude session to another. Without it, you would have to re-explain everything to Claude Code from scratch. Download it. Drop it in your workspace folder.

**First prompt structure (6:30)**

The first prompt to Claude Code does four things in one message: requests a folder structure with [CLAUDE.md](http://claude.md/ "http://CLAUDE.md"), states the deployment target (GitHub Pages), sets scope boundaries (no user accounts, no complex features yet), and ends with "ask me three questions." Watch how much Claude processes before it asks anything. That processing is free. Rebuilding later is not.

**Asking before building (9:00)**

Claude asks alignment questions. These surface gaps you did not think to address. Logo files, form handling, API connections, social links. Each answer here prevents a round of rework later.

**The PRD step (10:30)**

Before any code gets written, Claude generates a Product Requirements Document using the UX/UI Design Skill. This includes design tokens, color values, typography, layout patterns, and a delivery checklist. The PRD is the blueprint. Review it. Edit it. This is your last free checkpoint.

**Project memory (13:00)**

Claude Code writes memory to a project file on your local machine. This happens automatically. It creates a memory index that references your project so it can pick up context across sessions. You do not need to build a separate memory system. The folder structure and [CLAUDE.md](http://claude.md/ "http://CLAUDE.md") handle routing. The built-in memory handles continuity.

**First build review (16:00)**

The first version looks wrong. Too dark, too empty, no photography. This is normal. The response is simple: take screenshots of what you do not like, describe what you want instead, and send it back. The second iteration is where the site starts looking right.

**What "I should have done this earlier" actually means (18:30)**

Every time you hear this phrase in the video, that is a token cost. Changing the color palette, not specifying image direction, skipping logo format details. All of these could have been in the first prompt or the PRD. The lesson is not that the process failed. The lesson is that planning saves money.

**The simple GitHub Pages route (20:00)**

Before the Claude Code push, the video shows the manual method. Create a repo, upload an index.html file, turn on Pages, get a URL. Three minutes. This is worth knowing even if you use the CLI method, because it helps you understand what GitHub Pages actually does.

**Pushing from Claude Code (23:00)**

Claude Code installs the GitHub CLI, handles authentication, creates the repo, sets up the.gitignore, and pushes. First-time setup takes a few minutes. After that, pushing changes is one prompt.

**CSS pathing issue (26:00)**

The first deploy broke the CSS because the file paths did not include the correct base URL prefix. The fix was telling Claude about the problem (or just pasting a screenshot). It recognized the issue, corrected the paths, and pushed again. This is a common deployment issue. It does not mean anything went wrong with the build. It means the local environment and the GitHub environment have different URL structures.

### What You Will Need

- A Claude account (Pro or above for Claude Code access)
- Claude Code installed (terminal or VS Code, see Lesson \[X\] for setup)
- A GitHub account (free at [github.com](http://github.com/ "http://github.com"))
- A website to build or rebuild (can be your own, a client's, or a reference site for practice)
- A folder on your desktop for the workspace

### Common Questions

**Do I need to know how to code?**

No. Claude Code writes the code. You need to know what you want and how to describe it clearly. If errors come up, you can paste them into Claude and it will fix them. Knowing basic HTML and CSS helps you understand what Claude is doing, but it is not required to follow this process.

**How much does this cost in tokens?**

The build in the video used under 10 prompts with Claude Code. Exact token count depends on the complexity of your site and how much rework you need. Planning well up front (Lesson 3.3) is the single biggest factor in keeping token costs low.

**Can I use this for a client project?**

Yes. The process is the same. The difference is that Steps 1 and 2 (analysis and briefing document) should happen during or after your discovery call with the client. The better your briefing document, the less revision you need later.

**What if I just want a simple one-page site?**

Skip Claude Code entirely. Build it as an artifact in Claude chat, download the HTML file, upload it to GitHub Pages manually. Three minutes. The full Claude Code process is for sites you plan to maintain, update, and expand over time.

**Can I change things after deployment?**

Yes. That is the entire point of the folder structure and GitHub setup. Open your folder, open Claude Code, describe the change, push it. The live site updates automatically. You can do this as often as you want.
