3.2 GitHub and Folder Structure

### \[Video Coming Soon\]What This Lesson Covers

In Lesson 3.1 you watched the full website build. This lesson slows down on two things that went by fast: how GitHub actually works and how folder structure changes what Claude Code can do.

These are connected. The folder structure you build locally is what GitHub stores and serves. If the folders are messy, Claude Code wastes tokens figuring out where things are. If the folders are clean, Claude reads a single [CLAUDE.md](http://claude.md/ "http://CLAUDE.md") file and knows exactly where to go.

---

### GitHub: What You Actually Need to Know

**Repositories.** A repo is a project folder that GitHub tracks. Every change you make gets recorded. You can always go back to an earlier version if something breaks. You will have one repo per website or project.

**Branches.** When you first create a repo, you have one branch called "main." This is the live version. You can create other branches to test changes without touching what is live. For most of what we do, you will work on main and push directly. Branching matters more when multiple people are working on the same project.

**Pushing.** When you push, you are sending your local files to GitHub. Claude Code does this for you. After the first setup (installing the GitHub CLI and authenticating), pushing is one prompt. "Push this to the repo." The live site updates automatically within about a minute.

**GitHub Pages.** This is the free hosting. GitHub looks at your repo, finds the website files, builds them, and serves them at a public URL. Two ways to set it up:

- Simple: upload an index.html file, turn on Pages in settings, select the main branch. Done.
- Astro/React/complex builds: GitHub Actions runs a build step automatically. Claude Code sets this up for you when you first push.

**GitHub Actions.** These are automated scripts that run when you push. For websites, the action builds your site from source files and deploys it. You will see a file called something like.github/workflows/deploy.yml in your repo. Claude Code creates this. You do not need to write it yourself but you should know it exists so you can troubleshoot if a deploy fails.

---

### The.gitignore File

This file lists everything that should stay on your local machine and never go to GitHub. Open it and read it before your first push.

What belongs in.gitignore:

- **.env files.** These hold API keys, passwords, and secrets. If you push a.env file to a public repo, those keys are visible to everyone on the internet. This is the most common security mistake beginners make.
- **node\_modules.** These are dependency files that rebuild automatically. They are large and do not need to be stored.
- **Build cache and dist folders.** Generated output. GitHub Actions rebuilds these during deployment.
- **Markdown files you want to keep private.** Your [CLAUDE.md](http://claude.md/ "http://CLAUDE.md"), brand voice docs, internal notes. These help Claude Code work but they do not need to be public. Add \*.md to your.gitignore if you want to exclude all markdown, or list specific filenames.

If you are not sure whether something should be in.gitignore, ask yourself: would I be comfortable if a stranger read this file? If the answer is no, add it.

### Folder Structure for Claude Code

The reason folder structure matters is routing. Claude Code reads your [CLAUDE.md](http://claude.md/ "http://CLAUDE.md") first. That file tells it where everything is and what each folder contains. If the [CLAUDE.md](http://claude.md/ "http://CLAUDE.md") is clear and the folders match what it describes, Claude spends zero tokens figuring out your project. It just starts working.

**The** [**CLAUDE.md**](http://claude.md/ "http://CLAUDE.md") **file**

This sits in the root of your project folder. It contains:

- What the project is
- Where to find key files (design tokens, brand voice, content, components)
- What the folder layout looks like
- Any rules or constraints (deployment target, tech stack, things to avoid)

Keep it short. Every token in [CLAUDE.md](http://claude.md/ "http://CLAUDE.md") gets read every time Claude Code loads your project. A 200-line [CLAUDE.md](http://claude.md/ "http://CLAUDE.md") costs you on every single prompt. Aim for 30 to 50 lines that cover the essentials and point to other files for detail.

**Folder-level README files**

Each major folder can have its own small markdown file explaining what is inside and why. Claude Code reads these when it enters that folder. This is how you give Claude detailed context without bloating the root [CLAUDE.md](http://claude.md/ "http://CLAUDE.md").

Example structure from the website build:

```
project-root/
  CLAUDE.md              (master routing file)
  docs/
    brand-voice.md       (tone, language rules, do/don't)
    design-tokens.md     (colors, fonts, spacing values)
    prd.md               (what we are building and why)
  src/
    components/
      README.md          (what components exist, how they connect)
    pages/
      README.md          (page list, routing logic)
    layouts/
      README.md          (layout templates, when to use each)
  public/
    images/
  .gitignore
```

**Why this works.** Claude Code navigates this the same way a new developer would navigate a well-organized codebase. It reads the top-level file, understands the structure, then goes to the specific folder it needs. No wasted tokens scanning through a giant flat directory of unnamed files.

**Multiple Claude instances.** Once this structure exists, you can open two or three Claude Code sessions pointed at the same folder. One works on components. Another writes content. Another researches images. They all read the same [CLAUDE.md](http://claude.md/ "http://CLAUDE.md") and stay coordinated. The folder structure is what makes this possible. Without it, multiple instances would overwrite each other or duplicate work.

### Action Steps

1. If you do not have a GitHub account, create one at [github.com](http://github.com/ "http://github.com")
2. Try the simple route first. Create a repo, upload any index.html file, turn on Pages, confirm you get a live URL
3. Open the.gitignore file from your Lesson 3.1 project (or any Claude Code project). Read every line. Do you know what each entry does?
4. Review your [CLAUDE.md](http://claude.md/ "http://CLAUDE.md"). Is it under 50 lines? Does it accurately describe your folder layout? If not, trim it
5. Post your GitHub Pages URL in the community
