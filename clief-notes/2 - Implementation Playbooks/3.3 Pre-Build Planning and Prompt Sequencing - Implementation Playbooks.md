3.3 Pre-Build Planning and Prompt Sequencing

### \[Video Coming Soon\]

### What This Lesson Covers

In Lesson 3.1 the planning phase went by in about ten minutes. This lesson is entirely about that phase because it applies to everything you build with Claude Code, not just websites. The method is the same whether you are building a website, an internal tool, a content system, or anything else.

The core idea: spend your thinking before you spend your tokens.

---

### The Planning Sequence

There are five steps. They happen before Claude Code creates a single file.

**Step 1: Analyze what exists**

If you are building from scratch, skip to Step 2. If you are rebuilding, referencing, or improving something that already exists, start in Claude chat (not Code) and give it the existing thing to study.

For a website, paste the URL and tell Claude to explore. For a document, upload it. For a process, describe it. The goal is to get Claude loaded with context about the current state before you ask it to build the next state.

Prompt pattern: "Look at this \[URL / document / description\]. Explore it thoroughly. Tell me what you find about its structure, content, and purpose."

**Step 2: Create a briefing document**

This is the move that saves the most tokens long term. Ask Claude to write a markdown file that describes what it found. Be explicit about who will read this document.

Prompt pattern: "Based on that, create a markdown file describing the structure and build. My goal is to give this to Claude Code. Write it for Claude to read, not for me."

Why this matters: Claude chat and Claude Code are separate sessions. They do not share memory. The markdown file is how you transfer context between them. Without it, you have to re-explain everything in your first Claude Code prompt, which is slow, expensive, and usually incomplete.

Download the markdown file. Drop it in your project folder.

**Step 3: Write the first prompt with boundaries**

Your first prompt to Claude Code does several jobs at once. Each one prevents a specific kind of waste.

- **Folder structure request.** Tell Claude what kind of organization you want, including a [CLAUDE.md](http://claude.md/ "http://CLAUDE.md"). This prevents it from dumping everything in one flat directory.
- **Deployment target.** State where this will be hosted (GitHub Pages, Vercel, a server, wherever). This prevents Claude from building complex deployment pipelines you do not need.
- **Scope boundaries.** Say what you are building now and what you are not building yet. "We do not need user accounts or authentication. We need a template that can support those later." This prevents Claude from over-engineering the first version.
- **Technology preferences.** If you have a preference (React, Astro, plain HTML), say it. If you do not, say "let us decide together" and let Claude recommend something.
- **The alignment question.** End with "ask me three questions to understand more." This is the most important line in the prompt. It forces Claude to read everything you provided, process it, and surface any gaps before it builds anything.

**Step 4: Answer the questions**

Claude will ask about things you did not think to specify. Logo files, API integrations, form handling, content management. Answer clearly. If you do not have an answer, say so and let Claude make a reasonable choice.

These questions are cheap. Answering them costs a few tokens. Discovering the same gaps after Claude has already built 15 files costs hundreds of tokens in rework.

**Step 5: Request the PRD before any code**

Ask Claude to create a Product Requirements Document. This is a markdown file that describes what will be built, how it will be organized, and what design decisions have been made.

Prompt pattern: "Based on all of that, create a PRD file for this project. Do not start building yet."

Review the PRD. Edit it if something is wrong. This is your last checkpoint before code starts flowing. Changes here are free. Changes after the build are expensive.

---

### Why This Works

The pattern is simple. You are front-loading the thinking.

Most people open Claude Code and start asking for things. Each prompt adds a little more context. By prompt 15 they have a mess of files, half of which do not match what they actually wanted, and they have burned through tokens fixing problems that should not have existed.

This method uses 4 to 5 prompts before any files get created. By the time Claude starts building, it knows the project structure, the design direction, the deployment target, and the scope boundaries. The build itself usually takes 2 to 3 prompts. Total prompt count for a full project: under 10.

Fewer prompts. Better output. Lower token cost. That is the whole thing.

---

### Applying This Beyond Websites

The sequence works for anything:

**Internal tools.** Analyze the current workflow (Step 1). Document it in markdown (Step 2). Describe what you want built, where it will run, and what it does not need to do yet (Step 3). Let Claude ask questions (Step 4). Get the PRD (Step 5). Build.

**Content systems.** Analyze existing content structure. Document the taxonomy and formats. Describe the output you want. Alignment questions. PRD. Build.

**Client work.** This is where the method pays for itself the most. Steps 1 and 2 happen during your discovery call. You walk away with a markdown briefing document. Steps 3 through 5 happen before you show the client anything. By the time they see the first build, it is already close to what they want because you planned it properly.

### Action Steps

1. Pick a project you want to build (does not have to be a website)
2. Run Steps 1 and 2 in Claude chat. Save the markdown file
3. Write your Step 3 prompt. Do not run it yet. Post it in the community for feedback
4. Run the full sequence. Count your total prompts from start to finished build
5. Compare that prompt count to the last time you built something without this method
